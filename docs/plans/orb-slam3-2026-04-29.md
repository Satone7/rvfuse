# ORB-SLAM3 RVV Analysis — AITC Team Execution Plan

**Date**: 2026-04-29 | **Status**: Complete | **Batch**: orb-slam3

## Context

ORB-SLAM3 is a real-time visual SLAM system (UZ-SLAMLab, CVPR 2020/2021) cross-compiled for RISC-V. Phases 0 (cross-compile) and 1 (hardware perf profiling) are complete. The remaining work: RVV vectorization of profiled hotspots → BBV profiling → cross-platform gap analysis → consolidated report.

This plan follows the AITC workflow. All tasks execute **serially** (machine performance constraint). Each teammate works in an isolated git worktree and reuses the main repository's pre-built toolchain (QEMU, LLVM 22, sysroot, OpenCV, ORB-SLAM3 libs).

**T1 result**: OpenCV rebuilt with RVV. libopencv_imgproc.so: 8,137 RVV instructions (GaussianBlur fully vectorized). libopencv_features2d.so: 50 RVV instructions (FAST partial). 328 FAST-related vector ops (vmslt, vmsgt, vand, vmv) but the full Bresenham circle pixel comparison would benefit from hand-tuned RVV intrinsics. See rvv-patches/gaussian-blur/README.md and rvv-patches/fast-corner/README.md.

**T2 result**: ABANDONED. FAST corner detection is fully auto-vectorized by the OpenCV rebuild (1,525 RVV instructions in libopencv_features2d.so, vmslt.vx/vmsgt.vx in FAST_t<8>). No greenfield code needed.

## Task Details

### ET-1: OpenCV CV_SIMD_SCALABLE RVV Verification (sonnet) [BLOCKER for T1]

**Scope**: Verify whether OpenCV 4.10.0's existing universal intrinsic SIMD path (`CV_SIMD_SCALABLE` + `intrin_rvv_scalable.hpp`) auto-generates correct RVV instructions for the GaussianBlur hot functions when compiled with `-march=rv64gcv_zvl512b`. This determines whether T1 is greenfield RVV implementation or enablement + verification.

**Phases**:
1. Examine OpenCV's CMake configuration for RVV flags (`-march`, `DCPU_BASELINE=RVV`, `DCV_SIMD_SCALABLE=ON`)
2. Compile `smooth.simd.hpp` (specifically `hlineSmoothONa_yzy_a` and `vlineSmoothONa_yzy_a` for `uint8_t,ufixedpoint16`) as a standalone test unit with `-march=rv64gcv_zvl512b -O3`
3. Disassemble the result with `llvm-objdump -d` and check for RVV instructions (vle, vse, vadd, vwmulu, vnclip, etc.)
4. If RVV instructions present: quantify coverage of the hot inner loop; report that T1 scope changes to "enable + verify + tune"
5. If RVV instructions absent: investigate why (missing HAL flag? wrong template instantiation?), document findings, report that T1 proceeds as greenfield implementation
6. Write findings to `applications/orb-slam3/rvv-patches/gaussian-blur/verification.md`

**Output**: `applications/orb-slam3/rvv-patches/gaussian-blur/verification.md` (findings + recommendation for T1 scope)

**References**: OpenCV source at `output/opencv/vendor/opencv/`, LLVM 22 at `third_party/llvm-install/`

### T1: OpenCV RVV Rebuild + OpenCV Operator Verification (sonnet)

**Scope**: Rebuild OpenCV 4.10.0 with `-DCPU_BASELINE=RVV -DRISCV_RVV_SCALABLE=ON` to enable the existing CV_SIMD_SCALABLE path, then verify that GaussianBlur and FAST corner detection generate correct RVV instructions. This replaces the original greenfield RVV implementation plan since ET-1 proved the universal intrinsics path already works.

**Prerequisite**: ET-1 verification report at `applications/orb-slam3/rvv-patches/gaussian-blur/verification.md`

**Phases**:
1. Rebuild OpenCV with `-DCPU_BASELINE=RVV -DRISCV_RVV_SCALABLE=ON -DCMAKE_C_FLAGS="-march=rv64gcv_zvl512b" -DCMAKE_CXX_FLAGS="-march=rv64gcv_zvl512b"`
2. Disassemble rebuilt `libopencv_imgproc.so` and confirm RVV instructions in:
   - `vlineSmoothONa_yzy_a<uint8_t,ufixedpoint16>` (GaussianBlur vline)
   - `hlineSmoothONa_yzy_a<uint8_t,ufixedpoint16>` (GaussianBlur hline)
   - FAST corner detection functions in `libopencv_features2d.so`
3. Document: RVV instruction counts per function, comparison vs ET-1 scalar baseline
4. If GaussianBlur verified: write `rvv-patches/gaussian-blur/README.md` with rebuild instructions + RVV coverage
5. If FAST corner also vectorized: write `rvv-patches/fast-corner/README.md` noting auto-vectorization; mark T2 as [-](abandoned/superseded)
6. If FAST corner NOT vectorized: document gaps, T2 proceeds as greenfield implementation
7. QEMU smoke test: run ORB-SLAM3 with rebuilt OpenCV, verify correct output

**Output**: Rebuilt OpenCV libs, `rvv-patches/gaussian-blur/README.md`, `rvv-patches/fast-corner/README.md` (or gap doc)

**References**: ET-1 verification.md, OpenCV build.sh, QEMU smoke test config

### T2: FAST Corner Detection RVV (sonnet)

**Conditional**: Only execute if T1 rebuild does NOT auto-vectorize FAST corner. If T1 covers FAST, this task is abandoned.

**Scope**: Implement RVV512 vectorization for OpenCV's FAST corner detector — Bresenham circle pixel comparisons (integer intensity).

**Source files**:
- Primary: `output/opencv/opencv-4.10.0/modules/features2d/src/fast.cpp`
- Pattern: 16-pixel Bresenham circle, threshold comparisons, corner score computation
- Integer ALU ops — different from float-heavy NN operators

**Phases**:
1. Analyze scalar implementation in fast.cpp
2. Design RVV approach: vectorized circle pixel load + vmslt/vmsgt threshold comparison
3. Create `rvv-patches/fast-corner/fast_corner_rvv.inl`
4. Write `rvv-patches/fast-corner/test.cpp`
5. Create `rvv-patches/fast-corner/patch.diff`
6. Write `rvv-patches/fast-corner/README.md`

**Output**: `applications/orb-slam3/rvv-patches/fast-corner/` (4 files)

**References**: `rvv-op` skill, OpenCV fast.cpp, Phase 1 perf data

### T3: ORB Descriptor (BRIEF) RVV (sonnet)

**Scope**: Implement RVV512 vectorization for ORB-SLAM3's BRIEF descriptor computation — 256-bit bit-level SIMD.

**Source files**:

- Pattern: 256 pixel-pair intensity comparisons → bit packing → 32-byte descriptor
- Bit operations: `vxor`, `vmslt`, `vmandnot`, `vcpop`

**Phases**:

2. Design RVV approach: vectorized pair loads + comparisons + bit packing
3. Create `rvv-patches/orb-descriptor/orb_descriptor_rvv.inl`
4. Write `rvv-patches/orb-descriptor/test.cpp`
5. Create `rvv-patches/orb-descriptor/patch.diff`
6. Write `rvv-patches/orb-descriptor/README.md`

**Output**: `applications/orb-slam3/rvv-patches/orb-descriptor/` (4 files)

**References**: `rvv-op` skill, ORB_SLAM3 source, Phase 1 perf data

### T4: g2o Eigen Auto-vectorization Verification (sonnet)

**Scope**: Verify whether Eigen 3.4.0's auto-vectorization already covers the g2o 6x6 dense matrix operations that account for ~16% runtime. This is a verification gate: if Eigen auto-vectorizes well for RISC-V, document and skip RVV implementation. If not, create an emergent RVV task.

**Phases**:
1. Locate the g2o hot path from Phase 1 perf annotate data: `Eigen::internal::dense_assignment_loop` and `BlockSolver` functions
2. Disassemble `libg2o.so` for RISC-V: check if inner matrix loops use RVV instructions (vle, vse, vfmacc, etc.)
3. If auto-vectorization is present: document which operations are covered, estimate coverage %, write findings to `rvv-patches/g2o-eigen-6x6/README.md`, report T4 COMPLETE
4. If auto-vectorization is absent or partial: create an emergent task for manual RVV implementation of the uncovered operations, report T4 COMPLETE with follow-up task scope

**Output**: `applications/orb-slam3/rvv-patches/g2o-eigen-6x6/README.md` (+ emergent task if needed)

**References**: Phase 1 perf annotate data, `output/orb-slam3/lib/libg2o.so`, `output/eigen3/eigen-3.4.0/`, LLVM 22 objdump

### T5: BBV Profiling Per Operator (sonnet)

**Scope**: Run QEMU BBV profiling for each RVV-implemented operator to collect basic block vectors for gap analysis.

**Dependencies**: T1, T2, T3, T4 complete (need RVV implementations and g2o resolution)

**Phases**:
1. For each operator with RVV implementation (gaussian-blur, fast-corner, orb-descriptor, + g2o if emergent task created):
   a. Build ORB-SLAM3 linking RVV-patched operator for RISC-V (`-march=rv64gcv_zvl512b`)
   b. Run `qemu-riscv64 -cpu max,vlen=512 -plugin tools/bbv/libbbv.so,... -L output/sysroot` with test dataset
   c. Collect `.bb` + `.disas` output files
2. Use `qemu-bbv-usage` skill for function-scoped profiling configuration

**Output per operator**: `rvv-patches/<operator>/bbv.<pid>.bb` + `rvv-patches/<operator>/disas.txt`

**References**: `qemu-bbv-usage` skill, ORB-SLAM3 build.sh, Phase 0 QEMU smoke test configuration

### T6: Cross-platform Gap Analysis (sonnet)

**Scope**: For each operator with BBV data, run cross-platform vector ISA comparison.

**Dependencies**: T5 complete (need BBV data)

**Phases** (per operator):
1. Run `rvv-gap-analysis` skill: compare RVV implementation against x86 AVX2, ARM NEON, LoongArch LSX/LASX
2. Calculate BBV-weighted benefit scores for proposed new instructions
3. Generate per-operator gap analysis report (`.md` + `.pdf`)

**Unique angles** (vs batch2 neural network apps):
1. GaussianBlur: fixed-point saturating convolution — `vnclip` efficiency vs x86 `_mm_packus_epi16`/NEON `vqmovun`
2. FAST corner: integer pixel intensity ops — RVV integer ALU vs float-heavy NN
3. BRIEF descriptor: bit-level 256-bit popcount — `vcpop` vs x86 `_mm_popcnt_u64`/NEON `vcnt`+`vaddv`
4. g2o Eigen: sparse Cholesky — gather/scatter efficiency

**Output per operator**: `docs/report/orb-slam3/<operator>/gap-analysis.md` + `docs/report/orb-slam3/<operator>/gap-analysis.pdf`

**References**: `rvv-gap-analysis` skill, BBV .bb files, Intel/ARM/LoongArch vector ISA manuals
