# DFG LLVM 22 Upgrade: riscv_instrs.json Regeneration and ISA Update

**Date:** 2026-04-10
**Status:** Approved

## Context

The project's LLVM toolchain has been upgraded to LLVM 22.1.3. The existing `tools/dfg/riscv_instrs.json` (67,252 records) was generated from the previous LLVM version. It needs to be regenerated from LLVM 22 to get the complete instruction set including new additions.

The llvm-project submodule is at commit `e9846648fd61` (LLVM 22.1.3).

## Design Decisions

### 1. Two-File I Extension Structure

The I extension is split into two files:

**`rv64i.py` — Auto-generated by `gen_isadesc.py`**

Contains all standard RV64I instructions that LLVM tablegen knows about.
These instructions have either **no predicates** (base ISA) or **only `IsRV64`** (RV64-specific variants like `addw`, `ld`).

- ~63 instructions: standard ALU, loads, stores, branches, jumps, CSR, system instructions
- Regeneratable: just re-run `gen_isadesc.py --ext I`
- Source of truth: LLVM tablegen definitions

**`rv64i-pseudo.py` — Hand-written**

Contains 28 pseudo-instructions that LLVM tablegen does NOT emit as standalone instructions:

| Pseudo | Expands To | Why LLVM doesn't have it |
|--------|-----------|-------------------------|
| `mv rd, rs1` | `addi rd, rs1, 0` | Alias, not a real encoding |
| `nop` | `addi x0, x0, 0` | Alias |
| `call` | `auipc ra, ...; jalr ra, ...` | Multi-instruction pseudo |
| `tail` | `auipc ra, ...; jr ra` | Multi-instruction pseudo |
| `ret` | `jalr x0, ra, 0` | Alias |
| `j` | `jal x0, imm` | QEMU emits as alias |
| `li rd, imm` | `lui/addi` sequence | Alias |
| `la rd, sym` | `auipc + addi` | Alias |
| `not rd, rs` | `xori rs, -1` | Alias |
| `neg rd, rs` | `sub rd, x0, rs` | Alias |
| `negw rd, rs` | `subw rd, x0, rs` | Alias |
| `seqz rd, rs` | `sltiu rd, rs, 1` | Alias |
| `snez rd, rs` | `sltu rd, x0, rs` | Alias |
| `sltz rd, rs` | `slt rd, rs, x0` | Alias |
| `sgtz rd, rs` | `slt rd, x0, rs` | Alias |
| `bgtz`/`blez` | Branch aliases | Alias |
| `bgt`/`ble`/`bgtu`/`bleu` | Branch aliases | Alias |
| `beqz`/`bnez` | Branch aliases | Alias |
| `ecall` | System call | In auto-gen but has no register flow, overridden for consistency |
| `ebreak` | Breakpoint | No register operands, filtered from auto-gen |
| `fence` | Memory ordering | fencearg operands, not GPR, filtered from auto-gen |
| `fence.i` | Instruction fence | No register operands, filtered from auto-gen |

**Why these MUST be hand-written:**
- They don't exist in LLVM tablegen as separate instructions — they are assembler-level aliases that the assembler expands into real instructions, OR they are system instructions with no register operands that get filtered by gen_isadesc.py's "must have register flow" check
- QEMU disassembly output may emit these pseudo-mnemonics, so the DFG parser must recognize them
- Their register flow is derived from the underlying real instruction (e.g., `mv rd, rs1` has the same flow as `addi rd, rs1, 0`)

**Loading order in `__main__.py`:**
1. Load `rv64i.py` → registers standard I instructions
2. Load `rv64i-pseudo.py` → registers pseudo-instructions (may override or supplement)

### 2. I Extension Generation Logic

Since the I extension has no explicit `HasStdExtI` predicate in LLVM (it's the base ISA), the filtering logic is:

- Include instructions with **no predicates** OR **only `IsRV64`** predicate
- Exclude instructions starting with known LLVM internal prefixes (`Pseudo`, `SDT_`, `Select_`, `G_`, `COPY`, etc.)
- Exclude instructions with empty `AsmString` (internal-only)

### 3. F/M/V Extensions

F, M, V extensions continue to use the existing `HasStdExt{F,M,V}` predicate matching.
No changes to filtering logic — just regenerate with the new JSON.

### 4. New Instructions to Check

The regenerated JSON will include these new instructions not in the old version:
- `fence.tso` — RISC-V Zicfiss extension
- `fentry_call` — XRay instrumentation
- `.insn` family — raw instruction encoding
- Various CSR and system instructions (`csrrw`, `csrrs`, `csrrc`, etc.)
- `unimp`, `wfi`, `mret`, `sret`, `dret`
- `sfence.vma`

## Implementation Steps

1. **Build llvm-tblgen** from LLVM 22.1.3 source → `riscv_instrs.json` regenerated ✅ (already done)
2. **Modify `gen_isadesc.py`**: Add I extension support with empty-predicate filtering
3. **Generate `rv64i.py`** from new JSON using `gen_isadesc.py --ext I`
4. **Create `rv64i-pseudo.py`**: Extract 16 pseudo-instructions from old `rv64i.py`
5. **Regenerate `rv64f.py` and `rv64m.py`** from new JSON
6. **Regenerate `rv64v.py`** from new JSON
7. **Update `__main__.py`**: Load `rv64i` + `rv64i-pseudo` when `--isa I` is specified
8. **Run all tests** — ensure no regressions
9. **Update `setup_tblgen.sh`**: Document the I extension generation command
