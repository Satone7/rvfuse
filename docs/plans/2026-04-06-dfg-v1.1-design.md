# DFG Tool v1.1 Design

## Motivation

DFG generation pipeline (v1.0, PR #6) processes basic blocks serially, offers limited debug visibility, outputs flat file structures, and has no model selection for agent calls. This iteration adds parallelism, richer logging, organized output, and model configuration.

## New CLI Arguments

| Argument | Short | Type | Default | Description |
|----------|-------|------|---------|-------------|
| `--debug` | — | flag | `False` | Full-pipeline detailed logging (implies `--verbose`) |
| `--jobs` | `-j` | int | `1` | Number of parallel processes for BB handling |
| `--model` | `-m` | str | `None` | Model name passed to `claude --model` |

`--debug` and `--verbose` coexist: when both present, `--debug` takes precedence.

## Output Directory Structure

```
dfg/
├── dot/           # DOT source files
│   ├── bb_001.dot
│   └── bb_002.dot
├── json/          # JSON metadata files

├── png/           # Rendered PNG images
│   ├── bb_001.png
│   └── bb_002.png
```

## Feature Details

### 1. --debug: Full-Pipeline Detailed Logging

**Scope**: Every stage outputs per-BB structured detail.

| Stage | Debug output |
|-------|-------------|
| Parsing | Regex match results for each instruction line, BB boundary detection |
| DFG Build | Per-instruction register flow resolution, `last_writer` map state after each instruction, RAW edge creation rationale |
| Agent Call | Full prompt text, raw response, parsed JSON, elapsed time |
| File Output | File paths written, content size summary |

**Logging architecture**:
- Each BB log block begins with a separator line containing BB ID and timestamp
- Debug logs are written to rotating log files only (not console) to avoid noise
- Console continues at INFO level regardless of `--debug`
- Uses `logging.handlers.RotatingFileHandler`: `maxBytes=100_000_000` (100 MB), `backupCount=0` (unlimited rotation)
- Log file location: `<output_dir>/debug.log`, rotating to `debug.log.1`, `debug.log.2`, etc.

**When `--verbose` only** (no `--debug`): Console shows DEBUG-level messages as in v1.0, no file logging.

### 2. -j: Parallel BB Processing

**Implementation**: `concurrent.futures.ProcessPoolExecutor` + `rich.progress.Progress`

- Each worker executes the full per-BB pipeline: instruction check, DFG build, agent check/generate, file writing
- `-j 1` falls back to serial behavior (no progress bar, identical to v1.0)
- Progress bar displays: completed/total BB count, elapsed time, current BB ID
- Results (counters) are collected from workers via `futures.as_completed()` and aggregated in the main process
- New dependency: `rich` added to `pyproject.toml`

**Worker function design**:
```python
def process_single_bb(bb: BasicBlock, registry: ISARegistry,
                      agent: AgentDispatcher, output_dir: Path,
                      debug: bool) -> ProcessResult:
    # Returns a dataclass with: bb_id, source, agent_check_result, file_paths, debug_logs
```

**Key constraints**:
- `ProcessPoolExecutor` requires picklable arguments — `AgentDispatcher` must be reconstructed per worker (not shared)
- `ISARegistry` is read-only and safely shared via pickling
- File I/O happens in workers; no shared state between workers

### 3. DOT-to-PNG Auto-Conversion

**Conversion**: `subprocess.run(["dot", "-Tpng", "<input.dot>", "-o", "<output.png>"])`

**graphviz availability check**:
- Before conversion batch, run `shutil.which("dot")` to check availability
- If unavailable: print a warning (`WARNING: graphviz 'dot' not found — skipping PNG generation`) and skip all conversions
- Individual conversion failures logged as warnings, not errors — do not block pipeline

**Conversion timing**: After all DOT files are generated (post-parallel phase), convert in a batch loop.

### 4. --model: Agent Model Selection

**Implementation**:
- `AgentDispatcher.__init__` gains `model: str | None = None` parameter
- Stored as `self._model`
- When building the subprocess command, if `self._model` is set: `["claude", "--model", self._model, "--print", prompt]`
- Value is passed through directly to Claude Code CLI — no alias/mapping logic
