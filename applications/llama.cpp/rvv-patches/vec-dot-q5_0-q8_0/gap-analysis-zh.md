# RVV 差距分析：vec_dot_q5_0_q8_0

## 对比：x86 AVX2 vs ARM NEON vs RISC-V RVV

### 算法概述

Q5_0 点积需要四个步骤：
1. **半字节解包（Nibble unpack）**：从 16 个打包字节中提取 32 个 4-bit 值
2. **符号扩展（Sign extension）**：根据 qh 位掩码应用第 5 位（bit=0 处减去 0x10）
3. **乘累加（Multiply-accumulate）**：计算 int8 × int8 点积
4. **缩放（Scale）**：乘以组合 delta 因子

### 指令级对比

| 步骤 | x86 AVX2 | ARM NEON | RISC-V RVV |
|------|----------|----------|------------|
| **半字节解包** | `PSHUFB` + `VPSRLDQ` + `VPAND` | `VAND` + `VSHR` | `VAND.VI` + `VSRL.VI` |
| **qh 符号扩展** | `VPSHUFB` + `VPCMPEQB` + `VPANDNOT` + `VPOR` | 查表（`table_b2b_1`）+ `VSUB` | `VLM` + `VMNAND` + `VSUB.MU` |
| **乘法** | `VPMADDUBSW` + `VPMADDWD` | `VDOT.S32`（dotprod 扩展） | `VWMUL.VV` |
| **归约** | 水平求和（`VPHADDD`） | `VADDV` | `VWREDSUM.VS` |
| **融合乘加** | `VFMADD`（融合） | `VMLA`（标量） | `FMADD`（标量） |

### 寄存器压力与吞吐量

| 指标 | x86 AVX2 | ARM NEON | RVV（VLEN=512） | RVV（VLEN=128） |
|------|----------|----------|-----------------|-----------------|
| 寄存器宽度 | 256-bit YMM | 128-bit Q | 512-bit V | 128-bit V |
| 每次迭代元素数 | 32 | 64（2 个块） | 32（1 个块） | 32（1 个块） |
| 使用的 LMUL | 不适用 | 不适用 | m1（mf2 加载） | m2 |
| 掩码寄存器 | 不适用（隐式） | 不适用（隐式） | v0（b8） | v0（b4） |
| 累加器 | 1（标量） | 2（并行向量） | 1（标量） | 1（标量） |

### 关键差距：qh 符号扩展

qh 符号扩展步骤揭示了最显著的架构差异：

**x86 AVX2**（4 条指令）：
```asm
VPSHUFB  xmm_qh, shuf_mask     ; 通过 shuffle 将位广播到字节
VPCMPEQB bytes, all_ones         ; 比较以获取 0x00/0xFF 掩码
VPANDNOT qh_mask, 0xF0           ; 取反并掩码
VPOR     result, nibbles         ; 与半字节值进行或运算
```
- `VPSHUFB` 可使用 256 位 shuffle 掩码任意排列字节——极其灵活
- qh→字节扩展是通过预计算掩码实现的单指令操作

**ARM NEON**（查表 + 减法）：
```c
// 查表：每个 qh 字节索引 → 8 字节预计算模式
tmp[i] = table_b2b_1[(qh >> (i*8)) & 0xFF];  // 0 → 0x00, 1 → 0x10
qhl = vld1q_s8((const int8_t *)(tmp + 0));
result = vsubq_s8(nibbles, qhl);               // 单次减法
```
- 使用 **256 字节查找表**（2^8 个条目）——需要内存但速度极快
- 基于减法的方法很优雅：bit=1 时执行 `value - 0x10`，bit=0 时执行 `value - 0x00`
- 然而，查表加载增加了内存压力

**RISC-V RVV**（掩码减法——最接近 ARM 的方法）：
```c
vbool8_t qh = __riscv_vlm_v_b8(x->qh, vl);     // 将 32 位加载为掩码
qh = __riscv_vmnand_mm_b8(qh, qh, vl);          // 取反所有位
vint8m1_t v0f = __riscv_vsub_vx_i8m1_mu(qh, v0c, v0c, 0x10, vl);  // 掩码减法
```
- 使用 **向量按掩码加载**（`VLM`）——将按字节加载的 qh 直接转换为掩码寄存器
- **掩码减法**（`VSUB.MU`）——仅在掩码激活（取反后）处执行减法
- 无需查找表——完全基于寄存器
- 这实际上比 x86 和 ARM 的方法都**更简洁**

### 差距评估

#### RVV 优势（对比 x86 AVX2 / ARM NEON）

1. **可配置 VLEN**：单一代码路径即可处理 128/256/512 位——x86 需要独立的 SSE/AVX2 路径；ARM 需要特性检测
2. **基于掩码的符号扩展**：`VLM` + `VMNAND` + `VSUB.MU` 共 3 条指令，优于 AVX2 的 4 条。对比 ARM 没有查找表的内存开销
3. **可扩展的元素数量**：VLEN=512 使用 m2 可处理 64 个元素——吞吐量可能达到 AVX2（32 个元素）的 2 倍
4. **自然的半字节解包**：`VAND.VI` + `VSRL.VI` 与 ARM NEON 方法完全相同，效率相当

#### RVV 劣势 / 差距

1. **缺少融合乘减与符号提取**：x86 `VPMADDUBSW` 将符号提取 + 无符号×有符号乘法 + 累加合并为一条指令。RVV 需要独立的符号扩展 + `VWMUL.VV` + `VWREDSUM.VS`（3 条指令 vs 1 条）。

2. **缺少整数点积指令**：ARM `VDOT.S32` 在一条指令中计算 `sum(dp[i]*dq[i])`（4×int8→int32）。RVV `VWMUL.VV` 产生 int16 中间结果，需要独立的归约步骤。这需要 2 条指令 vs 1 条。

3. **归约瓶颈**：`VWREDSUM.VS` 要求标量累加器，无法并行累加。ARM NEON 使用 2 个向量累加器（sumv0、sumv1）以获得更好的流水线利用率。RVV 可以通过 `vslideup` + `vadd` 维护跨迭代的向量累加器来改善。

4. **缺少字节重排（PSHUFB 等价物）**：RVV 缺少通用的字节级重排指令。这限制了 x86 上可能实现的一些优化。最接近的是 `vrgather` 指令，但需要索引计算。

5. **跨 VLEN 的掩码寄存器类型不匹配**：VLEN=128 需要 `b4` 掩码类型；VLEN>=256 需要 `b8` 掩码类型。这要求运行时 VLEN 检测和双代码路径——是一种本可避免的复杂度。

### 面向 Q5_0 点积的 RVV 扩展建议

| 差距 | 建议扩展 | 收益 |
|------|---------|------|
| 无 int8 点积 | `vdot.s8`（4×i8→i32） | 用单条指令替代 VWMUL+VWREDSUM；与 ARM VDOT 对齐 |
| 掩码乘减 | `vmsub.vx`（掩码标量减法） | 将 VMNAND+VSUB.MU 合并为单条符号扩展指令 |
| 可配置掩码类型 | 根据目标 LMUL 自动提升 b4↔b8 | 消除掩码操作的运行时 VLEN 分支 |
| 跨迭代向量累加 | `vredsum.vs` 支持向量（非标量）源 | 支持类似 ARM NEON 双累加器的多迭代向量累加 |

### 指令数对比（每个 Q5_0 块）

| 步骤 | x86 AVX2 | ARM NEON | RVV VLEN=512 | RVV VLEN=128 |
|------|----------|----------|-------------|-------------|
| 半字节解包 | 3 | 2 | 4 | 3 |
| qh 符号扩展 | 4 | 2（查表） | 3 | 3 |
| 乘法 | 1（PMADDUBSW） | 1（VDOT） | 1（VWMUL） | 1（VWMUL） |
| 归约 | 1（PMADDWD） | 1（VADDV） | 2（VWREDSUM+MV） | 2（VWREDSUM+MV） |
| 缩放 | 2（FP16+FMUL） | 2（FP16+VMLA） | 2（FP16+FMUL） | 2（FP16+FMUL） |
| **合计** | **11** | **8** | **12** | **11** |

### 吞吐量对比（每条指令处理的元素数）

| 平台 | 每次迭代元素数 | 每次迭代指令数 | 吞吐量（元素/指令） |
|------|--------------|----------------|-------------------|
| x86 AVX2 | 32 | 11 | 2.91 |
| ARM NEON | 64 | 8 | 8.00 |
| RVV VLEN=512 | 32 | 12 | 2.67 |
| RVV VLEN=128 | 32 | 11 | 2.91 |

ARM NEON 由于 `VDOT.S32`（乘法+归约仅需 1 条指令）和双块处理（每次迭代 64 个元素）而领先。RVV 通过增加 `vdot.s8` 扩展和多块迭代可以达到甚至超越此水平。

## 性能收益估算

> **数据来源**: Spacemit K1-X (rv64imafdcv) 性能热点分析，Qwen2.5-0.5B Q4_K_M 模型，生成 128 tokens。
> **指令延迟**: K1-X RVV 流水线实测值（参见 `docs/reference/cx/instruction-constraints-and-latency.md`）。

### 估算方法

估算分三步进行：

1. **周期级 kernel 分析**：使用 K1-X 流水线的实测指令延迟，计算标量通用实现与 RVV 实现的关键路径周期数。
2. **Kernel 加速比推导**：标量 vs RVV 每 Q5_0 块（32 元素）的周期比。
3. **端到端推算**：使用性能分析测量的 Q5_0 时间占比（55.45%），应用 Amdahl 定律推算整体推理加速比。

所有周期数均为**计算密集型**估算（假设数据在 L1/L2 缓存中）。单独提供内存受限分析。

### K1-X 指令延迟参考

| 指令 | 延迟（周期） | 说明 |
|------|-------------|------|
| `vsetvli` | 2 | 循环设置（可摊销） |
| `vle8.v`（向量加载） | 3 | 16 字节 (m1) 或 32 字节 (m2) |
| `vlm.v`（掩码加载） | 3 | 4 字节 → 掩码寄存器 |
| `vand.vi` | 3 | 立即数操作数 |
| `vsrl.vi` | 3 | 立即数操作数 |
| `vmnand.mm` | 3 | 掩码操作 |
| `vsub.vx`（掩码版） | 4+3 = 7 | 标量广播额外开销 |
| `vwmul.vv`（SEW=8） | 4 | int8×int8 → int16 |
| `vwredsum.vs` | 4 | int16 → int32 归约 |
| `vmv.x.s` | 3 | 向量 → 标量移动 |
| `lb`（标量字节加载） | 3 | 标量基线 |
| `mul`（标量乘法） | 5 | RVG 基线 |

### 标量通用实现：周期分析（每个 Q5_0 块，32 元素）

通用实现循环 16 次（qk/2），每次处理 2 个元素：

```c
for (int j = 0; j < 16; ++j) {
    xh_0 = ((qh >> j) & 1) << 4;        // 位提取：~2 周期
    xh_1 = ((qh >> (j+16)) & 1);        // 位提取：~2 周期
    x0 = (qs[j] & 0x0F) | xh_0) - 16;   // 加载+掩码+或+减：~7 周期
    x1 = ((qs[j] >> 4) | xh_1) - 16;    // 移位+或+减：~4 周期
    sumi0 += x0 * y[j];                  // 加载+乘+加：~9 周期
    sumi1 += x1 * y[j+16];               // 加载+乘+加：~9 周期
}
```

**每次迭代关键路径**（顺序执行，保守估计）：
- 乘法数据就绪：~7 周期（加载 + 半字节操作）
- `mul`（5 周期）——瓶颈操作，通过 `add` 形成循环携带依赖
- 累加：1 周期
- **每次迭代：~13 周期**

**每个块总计**：16 × 13 + memcpy(3) + FP 缩放(8) = **~219 周期**

标量乘法（`mul`）5 周期延迟是最大的瓶颈——每次迭代出现两次，且在顺序执行核心上无法流水线化。

### RVV 实现：周期分析（每个 Q5_0 块，VLEN=128）

K1-X 延迟下的依赖图：

```
t0:  vle8.qs(3)          ─┬─→ vand(3)=6  ─┐
                           └─→ vsrl(3)=6  ─┤
t1:  vlm.qh(3)           ──→ vmnand(3)=6  ┤
t2:  vle8.y(3)           （独立）          │
t3:  vmv.zero(3)          （独立）         │
                                             ↓
t4:  max(6,6)+vcreate(~3)=9  ──────────→ vsub_mu(7)=16
t5:  max(16,3)+vwmul(4)=20  ←─────────── (y 在 3 就绪)
t6:  max(20,3)+vwredsum(4)=24
t7:  t6+vmv_x_s(3)=27
```

**RVV 每块关键路径：27 周期**（+ ~8 FP16 缩放 + 循环开销 = **~35 周期总计**）

### Kernel 加速比估算

| 实现 | 周期/块（32 元素） | 相对标量加速比 |
|------|-------------------|---------------|
| 标量通用 | ~219 | 1.0×（基线） |
| RVV (VLEN=128) | ~35 | **6.3×** |
| RVV (VLEN=256) | ~33 | **6.6×** |
| RVV (VLEN=512) | ~31 | **7.1×** |

加速的主要驱动力是将 16 次标量 `mul`（每次 5 周期）替换为单条 `vwmul.vv`（4 周期），一次性处理全部 32 个元素。标量版本仅乘法就消耗 16×2×5 = 160 周期；RVV 仅需 4 周期。

### 端到端推理影响（Amdahl 定律）

性能分析显示，`ggml_vec_dot_q5_0_q8_0_generic` 在 K1-X 上占推理总时间的 **55.45%**（Qwen2.5-0.5B Q4_K_M，128 tokens）。

Amdahl 定律：`总加速比 = 1 / [(1-f) + f/s]`，其中 f=0.5545，s=kernel 加速比。

| 场景 | Kernel 加速比 | Q5_0 时间占比 | 整体推理加速比 |
|------|-------------|--------------|--------------|
| 标量基线 | 1.0× | 55.45% | 1.00× |
| RVV VLEN=128（保守） | 5.0× | 11.09% | **1.80×** |
| RVV VLEN=128（乐观） | 6.3× | 8.80% | 1.91× |
| RVV VLEN=256 | 5.5× | 10.08% | 1.86× |
| RVV VLEN=512 | 6.0× | 9.24% | 1.89× |

**现实估计：Q5_0 RVV 优化单独可带来 ~1.8× 整体推理加速**。此估算假设为计算密集型操作（数据适合缓存），与性能分析中观察到的长标量执行时间一致。

### 扩展收益估算

#### 1. `vdot.s8` — 整数点积（最高影响）

替代 `vwmul.vv`(4) + `vwredsum.vs`(4) + `vmv.x.s`(3) = **11 周期**的单条指令。

预估 `vdot.s8` 延迟：**5 周期**（与 `vwmacc.vv` = 5 周期相当，两者都执行乘加操作）。

| 指标 | 优化前 | 优化后 | 改善幅度 |
|------|--------|--------|---------|
| 乘法-归约阶段 | 11 周期 | 5 周期 | **减少 54%** |
| RVV 总关键路径 | 27 周期 | 21 周期 | **加速 22%** |
| Kernel 相对标量加速比 | 6.3× | 8.1× | **+29%** |
| 整体推理加速比 (VLEN=128) | 1.80× | 1.89× | **+5%** |

相对较小的整体影响（+5%）是因为乘法-归约阶段（11 周期）仅占总流水线（27 周期）的一部分，且 Q5_0 kernel 本身仅占推理总时间的 55%。

#### 2. 掩码乘减 `vmulsub.vx`（中等影响）

替代 `vmnand.mm`(3) + `vsub.vx_mu`(7) = **10 周期**的单条指令。

预估延迟：**7 周期**（与 `vsub.vx` 掩码版相同，因为取反被吸收到指令内部）。

| 指标 | 优化前 | 优化后 | 改善幅度 |
|------|--------|--------|---------|
| 符号扩展阶段 | 10 周期 | 7 周期 | **减少 30%** |
| RVV 总关键路径 | 27 周期 | 24 周期 | **加速 11%** |
| Kernel 相对标量加速比 | 6.3× | 7.0× | **+11%** |
| 整体推理加速比 (VLEN=128) | 1.80× | 1.84× | **+2%** |

#### 3. 跨迭代向量累加器（流水线改善）

当前实现使用标量累加器（`vwredsum.vs` → `vmv.x.s`），导致迭代串行化。添加向量累加器（类似 ARM NEON 的双 `sumv0`/`sumv1` 模式）可以重叠迭代流水线，隐藏加载和设置延迟。

预估收益：通过隐藏跨迭代的加载延迟，**吞吐量提升 10-15%**。

#### 4. 可配置掩码类型（无计算收益）

消除依赖 VLEN 的代码路径分支（b4 vs b8）。有利于代码可维护性和减少二进制体积，但对特定 VLEN 配置 **无周期级收益**。

### 扩展组合场景

若所有提议的扩展全部实现：

| 配置 | 周期/块 | Kernel 加速比 | 整体推理加速比 |
|------|---------|-------------|--------------|
| 仅 RVV (VLEN=128) | ~35 | 6.3× | 1.80× |
| RVV + `vdot.s8` | ~29 | 7.6× | 1.86× |
| RVV + `vdot.s8` + `vmulsub` | ~26 | 8.4× | 1.89× |
| 全部扩展 + 多块迭代 | ~22 | 10.0× | 1.93× |

**Q5_0 kernel 优化单独可达到的最大整体推理加速比：~1.9×**

要突破此上限，需要向量化其余 vec_dot kernel：
- Q6_K × Q8_K (18.22%)：额外增加 ~0.3× 整体加速
- Q4_K × Q8_K (15.92%)：额外增加 ~0.2× 整体加速
- 全部 4 个 kernel 向量化后：**预估 ~2.5-3.0× 整体推理加速比**

### 内存受限上限

每个 Q5_0 块访问 56 字节（16B qs + 4B qh + 2B d + 32B y + 2B d）。K1-X L2 带宽约 20 GB/s，理论每块最小耗时 56/20 ≈ 2.8 ns。在 1.8 GHz 下约为 5 个周期——远低于计算受限的 27 周期估算。**该 kernel 是明确的计算密集型**，确认指令级优化（扩展）确实能带来实际加速。

### 总结

RVV 的 `vec_dot_q5_0_q8_0` 实现在功能上正确，指令数与 x86 AVX2 相当。基于掩码的符号扩展方法（`VLM`+`VMNAND`+`VSUB.MU`）简洁优雅，避免了 ARM NEON 查找表的内存开销。

基于 K1-X 指令延迟分析，RVV 实现相对标量通用基线达到 **6.3× kernel 加速比**，对 Q5_0 占主导的 Q4_K_M 工作负载可带来 **~1.8× 整体推理加速**。提议的 `vdot.s8` 扩展可将此提升至 **~1.9×**——有一定意义但增量有限。最大的优化空间在于向量化其余三个 vec_dot kernel（Q6_K、Q4_K、Q8_0），以达到 **~2.5-3.0×** 总推理加速。
